# Manual MVP test checklist

Этот чеклист описывает ручную end-to-end проверку MVP Telegram-бота AI Manager для SLIK Place. Проверку удобно выполнять с двумя Telegram-аккаунтами: `Owner` и `Employee`.

## Preconditions

- Установлен Node.js 20+.
- Есть Telegram-бот и его `BOT_TOKEN` от BotFather.
- Известен Telegram ID аккаунта владельца.
- Есть второй Telegram-аккаунт для проверки роли сотрудника.
- Репозиторий склонирован локально.

## 1. Запуск проекта

1. Перейти в корень репозитория.
2. Установить зависимости:

   ```bash
   npm install
   ```

3. Сгенерировать Prisma Client:

   ```bash
   npx prisma generate
   ```

4. Применить миграции:

   ```bash
   npx prisma migrate dev
   ```

5. Запустить бота в режиме разработки:

   ```bash
   npm run dev
   ```

6. Убедиться, что процесс стартовал без ошибок и бот отвечает в Telegram.

## 2. Настройка `.env`

1. Скопировать пример конфигурации:

   ```bash
   cp .env.example .env
   ```

2. Заполнить обязательные значения:

   ```dotenv
   BOT_TOKEN=123456:replace_with_real_telegram_bot_token
   DATABASE_URL="file:./data/dev.sqlite"
   OWNER_TELEGRAM_IDS=123456789
   ```

3. Проверить, что `OWNER_TELEGRAM_IDS` содержит Telegram ID аккаунта владельца.
4. Если владельцев несколько, указать ID через запятую без пробелов.
5. Перезапустить бота после изменения `.env`.

## 3. Регистрация Owner

1. Открыть бота в Telegram под аккаунтом Owner.
2. Отправить команду:

   ```text
   /start
   ```

3. Ожидаемый результат:
   - бот регистрирует пользователя;
   - пользователь получает роль `OWNER`;
   - пользователь активен без ручного подтверждения;
   - меню содержит команды владельца.
4. Проверить профиль:

   ```text
   /me
   ```

5. Убедиться, что в ответе отображается роль `OWNER` и активный статус.

## 4. Регистрация Employee

1. Открыть бота в Telegram под аккаунтом Employee.
2. Отправить команду:

   ```text
   /start
   ```

3. Ожидаемый результат:
   - бот регистрирует пользователя;
   - пользователь получает роль `EMPLOYEE`;
   - статус пользователя — `PENDING`;
   - до подтверждения сотруднику недоступны рабочие команды.
4. Сохранить Telegram ID сотрудника из ответа бота или из профиля:

   ```text
   /me
   ```

## 5. Approve employee

1. Под аккаунтом Owner выполнить команду:

   ```text
   /approve <employeeTelegramId>
   ```

2. Ожидаемый результат:
   - бот подтверждает сотрудника;
   - статус Employee становится `ACTIVE`.
3. Под аккаунтом Employee выполнить:

   ```text
   /me
   ```

4. Убедиться, что статус сотрудника активный.

## 6. Create shift

1. Под аккаунтом Owner создать смену:

   ```text
   /create_shift 2026-07-05 18:00 23:00 Общий зал
   ```

2. Ожидаемый результат:
   - бот создает смену;
   - смена получает статус `OPEN`;
   - в ответе отображается `shiftId`.
3. Сохранить `shiftId` для следующих шагов.
4. Проверить список смен:

   ```text
   /shifts
   ```

5. Убедиться, что новая смена отображается у Owner.

## 7. Take shift

1. Под аккаунтом Employee открыть доступные смены:

   ```text
   /shifts
   ```

2. Убедиться, что созданная смена отображается в списке доступных.
3. Откликнуться на смену:

   ```text
   /take_shift <shiftId>
   ```

4. Ожидаемый результат:
   - бот принимает отклик `TAKE`;
   - смена остается доступной до назначения владельцем.

## 8. Assign shift

1. Под аккаунтом Owner посмотреть отклики:

   ```text
   /shift_responses <shiftId>
   ```

2. Убедиться, что отклик Employee отображается в списке.
3. Назначить сотрудника на смену:

   ```text
   /assign_shift <shiftId> <employeeTelegramId>
   ```

4. Ожидаемый результат:
   - смена назначена Employee;
   - статус смены становится `ASSIGNED`.
5. Под аккаунтом Employee проверить назначенные смены:

   ```text
   /my_shifts
   ```

6. Убедиться, что смена отображается как назначенная.

## 9. Start shift with photo

1. Под аккаунтом Employee выполнить:

   ```text
   /start_shift <shiftId>
   ```

2. Ожидаемый результат: бот просит отправить фото начала смены.
3. Отправить фотографию в Telegram-чат с ботом.
4. Ожидаемый результат:
   - фото сохранено как `START`;
   - статус смены становится `STARTED`;
   - бот подтверждает начало смены.

## 10. Ready shift with photo

1. Под аккаунтом Employee выполнить:

   ```text
   /ready_shift <shiftId>
   ```

2. Ожидаемый результат: бот просит отправить фото готовности площадки.
3. Отправить фотографию в Telegram-чат с ботом.
4. Ожидаемый результат:
   - фото сохранено как `READY`;
   - статус смены становится `READY`;
   - бот подтверждает готовность площадки.

## 11. End shift with photo

1. Под аккаунтом Employee выполнить:

   ```text
   /end_shift <shiftId>
   ```

2. Ожидаемый результат: бот просит отправить фото окончания смены.
3. Отправить фотографию в Telegram-чат с ботом.
4. Ожидаемый результат:
   - фото сохранено как `END`;
   - статус смены становится `COMPLETED`;
   - бот подтверждает завершение смены.

## 12. Report shift

1. Под аккаунтом Employee выполнить:

   ```text
   /report_shift <shiftId>
   ```

2. Последовательно ответить на вопросы бота:
   - количество гостей;
   - были ли проблемы;
   - были ли повреждения;
   - был ли конфликт;
   - комментарий по смене.
3. Ожидаемый результат:
   - отчет сохранен;
   - статус смены становится `CLOSED`.
4. Под аккаунтом Owner проверить отчет:

   ```text
   /shift_report <shiftId>
   ```

5. Убедиться, что данные отчета отображаются корректно.

## 13. Set sales

1. Под аккаунтом Owner указать продажи по закрытой смене:

   ```text
   /set_sales <shiftId> 15000
   ```

2. Ожидаемый результат:
   - сумма продаж сохранена;
   - начисление по смене пересчитано с учетом продаж.

## 14. Check payroll

1. Под аккаунтом Owner открыть общий список начислений:

   ```text
   /payroll
   ```

2. Ожидаемый результат:
   - в списке есть начисление по проверяемой смене;
   - отображаются базовая сумма, продажи, бонус и итог.
3. Под аккаунтом Employee проверить личные начисления:

   ```text
   /my_payroll
   ```

4. Убедиться, что сотрудник видит начисление по своей закрытой смене.

## 15. Mark paid

1. Под аккаунтом Owner отметить выплату сотруднику:

   ```text
   /mark_paid <employeeTelegramId> <amount>
   ```

2. Ожидаемый результат:
   - бот создает запись о выплате;
   - подходящие начисления сотрудника получают статус `PAID`;
   - бот сообщает количество закрытых начислений.
3. Повторно проверить список начислений:

   ```text
   /payroll
   ```

4. Убедиться, что оплаченные начисления больше не отображаются как ожидающие выплаты.

## Known limitations

- Нет интеграции с YClients.
- Продажи пока вводятся вручную.
- Нет веб-панели.
- Нет автоматических уведомлений.
